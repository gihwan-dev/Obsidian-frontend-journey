# Outcome (1 line, result-first)
gitlab-slash-command에 reconcile 제어와 의도 분류/실행 라우팅 분리를 더해, AI 자동화가 질문을 수정 요청으로 오인하는 경로와 메인 스레드 병목을 줄일 운영 기준으로 확장했다.

## Context
- 자동 생성 이슈가 바로 후속 실행으로 이어지면 불필요한 작업이 누적되고 운영 리스크가 커질 수 있었다.
- 이슈 생성, 검증, 실행을 분리한 제어 지점이 있어야 AI 자동화를 협업 흐름에 안전하게 붙일 수 있었다.
- 멘션 기반 자동화에서 질문과 수정 요청을 같은 흐름으로 다루면 의도 오해로 잘못된 편집이 일어날 수 있었다.

## Actions
- 자동 생성 이슈를 다시 점검하고 후속 실행 여부를 가르는 reconcile 파이프라인을 추가했다.
- 웹훅 컨텍스트, 체크포인트, 작업 에이전트 도구를 분리해 이슈 처리 단계를 문서화하고 검증했다.
- 라벨 조건과 웹훅 대상을 구분해 자동화 범위가 과하게 확장되지 않도록 제어 기준을 정리했다.
- 질문과 수정 요청을 섞어 처리하지 않도록 의도 분류와 실행 라우팅을 분리하는 방향으로 라우팅 구조를 더 세분화했다.
- 메인 스레드 병목을 줄이기 위해 워커 풀과 큐잉 중심으로 동시성 도입 범위를 제한하고, 분산 시스템까지 넓히지 않는 구현 기준을 정리했다.
- 간헐적 실패 대응을 P1 이슈와 웹 리서치 도구 확장 방향까지 묶어 다음 작업 우선순위를 명확히 했다.

## Results
- 자동 생성 이슈를 바로 실행하지 않고 검증 후 연결하는 운영 제어 지점을 만들었다.
- 이슈 처리 흐름을 단계별로 재사용 가능한 구조로 남겨 후속 `/mr` 작업에도 같은 기준을 적용할 기반을 마련했다.
- 사용자의 질문을 수정 요청으로 오인해 자동화가 과하게 실행되는 경로를 줄이는 설계 기준을 더했다.
- 동시성 범위를 워커 풀과 큐잉 중심으로 한정해 메인 스레드 병목을 줄일 수 있는 구현 경계를 분명히 했다.
- 정량 효과는 아직 근거가 없어 후속 계측이 필요하다.
