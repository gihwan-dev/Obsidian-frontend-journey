# 과하고 무의미한 테스트

> AI는 테스트를 "행동 명세 검증"이 아니라 "작업 완료 증빙"으로 생산하는 경향이 있다.

## 현상

- 구현 세부에 의존하는 테스트 (내부 구조가 바뀌면 깨짐)
- 의미 없는 assertion (존재 확인만 하고 동작을 검증하지 않음)
- 하드코딩된 대기 시간 (`await wait(300)`)으로 비결정적 테스트 생성
- 같은 테스트를 반복 "개선"하지만 커버리지 의미는 변하지 않음

## 실제 사례

> exem-table 레포의 매일 오전 9시 테스트 자동화 리뷰

### 반복되는 "개선" — 실질은 없는 churning

매일 자동화로 AI가 테스트를 "정비"하는데, 커밋 메시지를 보면 패턴이 보인다:

```
b4372f4 test(react-table): strengthen table behavior coverage
4cf8ddf test(table): 테이블 동작 검증 테스트 보강
1311d6a test(react-table): 테이블 동작 중심 테스트 정비
62fe362 test(react-table): tighten table behavior coverage
cdfbd9a test(react-table): 행 고정 테스트를 동작 중심으로 정리
e6762ec test(react-table): 동작 중심 테이블 테스트 정리
0145e24 test(react-table): 테이블 동작 중심 테스트 정비
...
```

**같은 말의 변형이 반복된다.** "보강", "정리", "정비", "strengthen", "tighten", "refine" — 표현만 다르고 하는 일은 비슷하다.

### AI가 만든 원래 테스트의 문제

```ts
// Before: AI가 최초 생성한 테스트
const RENDER_WAIT_TIME = 300;

it('커스텀 아이콘이 렌더링된다', async () => {
  const { container } = render(<CustomIcons />);
  await wait(RENDER_WAIT_TIME);  // 하드코딩된 대기

  // "존재만 확인"하는 의미 없는 assertion
  const textContent = container.textContent || '';
  expect(textContent.includes('[+]') || textContent.includes('[-]')).toBe(true);
});
```

문제점:
- `await wait(300)` — 왜 300ms인가? 환경에 따라 불안정
- "아이콘이 렌더링된다"고 하면서 실제로는 텍스트 포함 여부만 체크
- 아이콘을 클릭했을 때 동작이 바뀌는지는 검증하지 않음

### 자동화가 "개선"한 버전

```ts
// After: 자동화가 수정한 테스트
it('커스텀 아이콘 스토리는 확장 상태에 따라 아이콘 텍스트가 바뀐다', async () => {
  const { container, 테스터 } = await 스토리를_렌더링하고_테스터를_준비한다(CustomIcons);
  const 사용자 = userEvent.setup();

  expect(container).toHaveTextContent('[-]');

  const Engineering_확장버튼 = 테스터.getExpanderInRow('Engineering');
  await 사용자.click(Engineering_확장버튼);

  await waitForRowPresence(테스터, 'Frontend Team', true);
  await waitFor(() => {
    expect(container).toHaveTextContent('[+]');
  });
});
```

개선된 점: `wait(300)` 제거, 실제 클릭 동작 검증 추가. 하지만 이건 **원래부터 이렇게 작성했어야 할 테스트**다. AI가 만든 문제를 AI가 고치는 순환이 반복된다.

### DEV 경고 테스트 → 비즈니스 계약 테스트로 교체

```ts
// Before: 구현 세부에 의존
it('size < minSize일 때 경고를 출력한다', () => {
  process.env.NODE_ENV = 'development';
  const warnSpy = vi.spyOn(console, 'warn').mockImplementation(() => {});
  createBaseColumn(makeColumn({ id: 'a', size: 36, minSize: 50 }));
  expect(warnSpy).toHaveBeenCalledWith(
    '[Table] 컬럼 "a": size(36)가 minSize(50)보다 작습니다...'
  );
});

// After: 동작 계약 검증
it('고정 크기 컬럼은 리사이즈 허용을 명시하지 않으면 비활성화한다', () => {
  const 결과 = createBaseColumn(makeColumn({ id: 'status', size: 80 }));
  expect(결과.enableResizing).toBe(false);
  expect(결과.meta?.hasExplicitSize).toBe(true);
});
```

console.warn 스파이에 의존하는 테스트는 경고 메시지 문구가 바뀌면 깨진다. 반면 "고정 크기면 리사이즈 비활성화"는 사용자 관점의 계약이다.

## 왜 이런 일이 발생하는가

1. **완료 증빙 편향**: 코딩 에이전트는 "테스트 통과 = 작업 완료"로 훈련됨. 테스트의 질보다 존재 자체가 강한 완료 신호
2. **표면 단서 의존**: LLM은 변수명, 코드 구조 같은 표면 단서로 테스트를 생성. 프로그램이 조금만 바뀌어도 의미 변화보다 구조 변화에 더 민감하게 반응
3. **오라클 문제**: "원래 이렇게 동작해야 한다"가 아니라 "지금 이렇게 동작한다"를 assert하는 경향. 명세가 약하면 현재 구현에 맞추는 회귀 오라클로 후퇴

## 판별 기준

- 이 테스트가 **구현을 바꿔도 동작이 같으면 통과하는가** → 좋은 테스트
- 이 테스트가 **내부 구조를 바꾸면 동작은 같은데 깨지는가** → 구현 결합 테스트
- 이 테스트가 **삭제해도 아무도 모르는가** → 무의미한 테스트
