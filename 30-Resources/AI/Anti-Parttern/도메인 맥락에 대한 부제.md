# 도메인 맥락에 대한 부재

> AI는 코드의 구문은 이해하지만, 그 코드가 존재하는 비즈니스 이유를 모른다.

## 현상

- API 스키마에 없는 기능을 "불필요"로 판단하고 삭제
- 자동 생성된 타입에 비즈니스 규칙을 결합해, 타입이 바뀌면 규칙도 같이 깨짐
- 코드에 명시되지 않은 도메인 지식(고객사 요구사항, 연계 규칙 등)을 고려하지 못함

## 실제 사례

> [maxgauge-dashboard MR #50](http://gitlab.exem.xyz/fe1/maxgauge-dashboard/-/merge_requests/50), [MR #52](http://gitlab.exem.xyz/fe1/maxgauge-dashboard/-/merge_requests/52) — Altibase 연계 기능 삭제 및 복구

### 사건 경위

1. Swagger 기반 API 자동 생성 스킬로 타입을 업데이트 시킴
2. 해당 시점의 Swagger 스키마에 Altibase 관련 정의가 일시적으로 빠져 있었음
3. AI가 스키마에 없는 Altibase를 "제거 대상"으로 판단, SSO 연계 로직에서 삭제
4. 고객사에서 Altibase SSO 로그인 불가 버그 발생

### 1차 수정 (MR #50) — 급한 불 끄기

```ts
// Altibase를 SSO 대상에 다시 추가
function isSSONavigatableDBType(dbType: DBType) {
  return dbType === DBType.ORACLE
    || dbType === DBType.TIBERO
    || dbType === DBType.ALTIBASE  // ← 복구
}
```

### 2차 수정 (MR #52) — 근본 원인 해결

```ts
// 비즈니스 규칙을 generated API 타입에서 분리
const SSO_NAVIGATABLE_DB_TYPES = ['ORACLE', 'TIBERO', 'ALTIBASE'];
const MFX_LOGIN_REQUIRED_DB_TYPES = ['MYSQL', 'SQLSERVER'];
```

핵심은 주석으로 남긴 이 한 줄이다:

> "연계 지원 DB 목록은 generated API enum/contract와 독립적인 비즈니스 규칙이다. API 타입에 특정 DB가 잠시 빠져 있더라도, 연계 규칙에서 임의로 제거하면 안 된다."

## 왜 이런 일이 발생하는가

1. **코드 = 진실이라는 가정**: AI는 현재 코드와 스키마를 유일한 진실 소스로 취급한다. 스키마에 없으면 없는 것이다.
2. **암묵지 접근 불가**: "Altibase 연계는 고객사 A가 쓰고 있다"는 정보는 코드 어디에도 적혀 있지 않다.
3. **일관성 편향**: API 타입과 비즈니스 로직의 불일치를 발견하면, AI는 "맞추는" 쪽으로 행동한다. 맞추면 안 되는 경우가 있다는 판단은 하지 못한다.

## 교훈

- 비즈니스 규칙과 자동 생성 코드는 물리적으로 분리해야 한다
- AI가 "정리"하거나 "동기화"한 변경은, 특히 삭제가 포함된 경우 반드시 도메인 지식을 가진 사람이 검토해야 한다
