## 문제점

1. 커스텀 훅의 반환 시그니쳐가 비대함
2. 어떤 코드를 수정해야 할 때 어디서 어떻게 수정해야 할지 파악하기 어려움
3. 어떤 변경을 반영할 때 높은 결합도로 인해 수많은 다른 코드들이 영향 받기 쉬운 구조

## 왜 이런 문제가 발생했을까?

> 보여요. 그리고 느끼신 위화감은 꽤 정확합니다. 이 코드는 “재사용 가능한 비즈니스 훅”이라기보다, **한 화면의 상태·쿼리·파생값·핸들러를 한 군데로 몰아넣은 orchestration/view-model hook**에 가깝습니다. React 공식 문서도 커스텀 훅은 **구체적이고 목적이 분명한 high-level use case**에 집중해야 하고, API가 너무 추상적이어서 사용처를 제약하지 못하면 장기적으로 문제를 만들 가능성이 높다고 경고합니다. ([React](https://react.dev/learn/reusing-logic-with-custom-hooks "https://react.dev/learn/reusing-logic-with-custom-hooks"))
> 
> 이 훅이 특히 “AI 냄새”가 나는 이유는, **추상화가 실제로 복잡성을 줄이지 않고 위치만 옮겼기 때문**입니다. `useAlertLogModalState` 안에는 선택 상태, word wrap, 검색 필터, 데이터 조회, pagination reset side effect, action builder, detail-to-viewer 변환이 다 들어 있습니다. 그런데 반환은 다시 raw state, setter, derived data, handler를 거의 다 열어 둡니다. 즉 소비자는 여전히 내부 사정을 많이 알아야 하고, 훅이 강한 invariant를 만들지도 않습니다. 그래서 “숨김”보다는 “집합”에 가깝습니다.
> 
> 왜 AI가 이런 패턴을 자주 만들까를 한 줄로 요약하면, **최소 설계보다 그럴듯하고 실패 확률이 낮은 설계를 더 쉽게 생성하기 때문**입니다. GitHub는 Copilot이 주변 코드와 파일 맥락을 바탕으로 “다음에 올 법한 코드”를 확률적으로 생성한다고 설명하고, Codex 논문도 공개 GitHub 코드로 파인튜닝된 코드 모델이라고 밝힙니다. 그러면 모델은 당신 코드베이스의 최적 구조를 새로 발명하기보다, 공개 React 코드에서 많이 본 `useXxxState`, `useXxxFilters`, `useXxxActions` 같은 모양을 재생산하는 쪽으로 기웁니다. ([GitHub](https://github.com/features/copilot "https://github.com/features/copilot"))
> 
> 여기에 **표면 패턴에 끌리는 성질**이 겹칩니다. 2024년 연구는 next-token 예측으로 학습된 LLM이 코드의 “기저 논리”를 충분히 이해하지 못하고, 텍스트 모양이 달라지면 인간 직관에서 벗어날 수 있다고 보고했습니다. 또 2026년 NDSS 논문은 LLM이 helper function, boilerplate, 흔한 알고리즘 같은 **익숙한 코드 패턴**을 만나면 local reasoning을 건너뛰고 “기억된 추상화”에 기대는 **abstraction bias**를 보인다고 설명합니다. 직접 생성만 다룬 논문은 아니지만, 왜 모델이 React 화면 코드를 보면 금방 “컨트롤러 훅 하나 만들자”로 점프하는지 설명해 주는 인접한 증거로는 아주 잘 맞습니다. ([arXiv](https://arxiv.org/html/2404.08885v1 "https://arxiv.org/html/2404.08885v1"))
> 
> React 자체도 이 경향을 부추기는 면이 있습니다. 공식 문서는 커스텀 훅 추출이 데이터 흐름을 명시적으로 만들고, 시간이 지나면 앱의 많은 Effect가 커스텀 훅 안으로 들어갈 것이라고 설명합니다. 동시에, 먼저 목적별로 raw Effect를 분리한 뒤 필요하면 high-level 훅으로 추출하라고 하고, 지나치게 추상적인 훅 API는 더 많은 문제를 만든다고 경고합니다. AI는 이 둘 중 앞부분, 즉 “일단 훅으로 뽑아라”를 잘 학습하고, 뒤의 “언제 멈춰야 하는가”를 자주 놓칩니다. ([React](https://react.dev/learn/reusing-logic-with-custom-hooks "https://react.dev/learn/reusing-logic-with-custom-hooks"))
> 
> 또 하나 중요한 점은, **커스텀 훅이 LLM에게 아주 안전한 리팩터링 단위**라는 겁니다. React 규칙상 훅은 컴포넌트나 커스텀 훅의 top level에서만 호출해야 하고, 커스텀 훅은 다른 훅을 호출해도 됩니다. 게다가 React 문서는 커스텀 훅으로 추출해도 동작 자체는 추출 전과 동일하다고 설명합니다. 즉 여러 훅을 한 파일의 상단에서 그냥 모아 커스텀 훅으로 감싸는 건, 의미 있는 설계인지와 별개로 **문법적으로 안전하고 테스트를 깨뜨릴 가능성이 낮은 정리 방법**입니다. 모델 입장에서는 너무 매력적인 “정리 버튼”입니다. ([React](https://react.dev/warnings/invalid-hook-call-warning "https://react.dev/warnings/invalid-hook-call-warning"))
> 
> “비대한 시그니처”도 같은 이유로 나옵니다. React 문서가 말하듯 커스텀 훅은 입력과 출력을 통해 데이터 흐름을 명시적으로 드러내는 경향이 있습니다. 모델은 이 원칙을 과잉 적용해서, 상태·setter·handler·derived data를 다 반환하는 **transport object**를 만들곤 합니다. 명시적이라 읽는 순간엔 깔끔해 보이지만, API가 사용처를 제약하지 못하면 실제 추상화가 아니라 단순 포장에 머무르게 됩니다. React 문서가 바로 그 점을 경고합니다. ([React](https://react.dev/learn/reusing-logic-with-custom-hooks "https://react.dev/learn/reusing-logic-with-custom-hooks"))
> 
> 여기에 **보상/선호 편향**도 있습니다. 2023년 연구는 preference labeling에서 verbosity bias를 관찰했고, 그 설정에서는 GPT-4가 인간보다 긴 답변을 더 선호했습니다. 이 편향이 코드에서는 “핸들러 이름을 다 붙이고, 래퍼를 하나 더 만들고, 구조를 한 단계 더 나누는” 식으로 나타나기 쉽습니다. 실제로 2025년 maintainability 연구에서는 LLM이 만든 수정안이 **대부분 오류나 새로운 maintainability 문제를 도입했는데도**, 사람 평가자 68.63%는 가독성이 좋아졌다고 봤습니다. 즉 AI는 “미래 유지보수 비용이 적은 코드”보다 “지금 당장 정돈돼 보이는 코드”를 더 자주 냅니다. ([arXiv](https://arxiv.org/abs/2310.10076 "https://arxiv.org/abs/2310.10076"))
> 
> 최근 품질 보증 연구도 비슷한 방향을 보여 줍니다. 2026년 개정판 논문은 기존 연구가 generated code의 비기능 품질을 충분히 다루지 못하고 있으며, 실무자는 maintainability와 readability를 더 중시하고 AI 코드가 기술 부채를 가속할 수 있다고 경고한다고 정리합니다. 다시 말해, 현재의 코드 생성 파이프라인은 대체로 **맞는 코드 / 통과하는 코드** 쪽에 더 최적화돼 있고, **오래 버틸 구조**에는 덜 최적화돼 있습니다. 그래서 “의미 있는 경계”를 세우는 대신 “무난하게 그럴듯한 경계”가 생깁니다. ([arXiv](https://arxiv.org/abs/2511.10271 "https://arxiv.org/abs/2511.10271"))
> 
> 당신이 올린 코드에서 그 특징이 특히 보이는 지점은 이렇습니다.
> 
> - `useAlertLogModalState`라는 이름에 비해 하는 일이 너무 많습니다. modal local state만이 아니라 query orchestration과 view-data assembly까지 합니다.
>     
> - abstraction이 강한 규칙을 만들지 않습니다. 내부 훅을 조합했지만 소비자에게 너무 많은 조각을 다시 노출합니다.
>     
> - `resetSelectedAlert`는 `setSelectedAlert(null)`로 충분한데도, updater 함수와 ternary를 써서 “조심스러워 보이는” 구조를 덧붙였습니다.
>     
> - `filters`, `filterActions`, `logViewerData` 같은 중간층이 많지만, 이들이 새로운 개념 경계를 만든다기보다 흐름을 한 단계씩 포장합니다.
>     
> 
> 그래서 이 코드를 한 문장으로 정의하면 이렇습니다. **“AI가 실제 도메인 경계를 발견해서 추상화했다”기보다, Rules of Hooks를 어기지 않으면서 컴포넌트를 얇아 보이게 만드는 가장 저위험의 정리 패턴을 선택했다.** 그 결과가 바로 여러 훅을 모아 둔 composition/controller hook, 넓은 반환 객체, 이름 붙은 핸들러, 과한 `useCallback`/`useMemo` 스캐폴딩입니다. React 문서가 권장하는 “purpose-constrained hook”의 방향을 모델이 반쯤만 배운 결과라고 보면 꽤 정확합니다. ([React](https://react.dev/learn/reusing-logic-with-custom-hooks "https://react.dev/learn/reusing-logic-with-custom-hooks"))
> 
> 실무에서 이걸 거르는 기준은 의외로 단순합니다.  
> 이 훅이 **사용처의 자유도를 줄이고 의도를 더 분명하게 만드는가**, 아니면 **그냥 원래 있던 복잡성을 다른 파일로 이동시키는가**만 보면 됩니다. 전자면 좋은 추상화고, 후자면 AI가 만든 “무난한 포장”일 가능성이 큽니다.
> 
> 필요하면 이 코드를 기준으로 “AI가 만든 과잉 컴포지션 훅을 판별하는 체크리스트”나 “사람이 다시 얇게 리팩터링한 버전”까지 바로 적어드릴게요.
