# 비대한 함수 입출력

> AI는 복잡성을 줄이는 대신, "정돈돼 보이는" 구조로 복잡성의 위치만 옮긴다.

## 현상

- 커스텀 hook이 상태, setter, 핸들러, 파생 데이터를 전부 반환하는 거대한 시그니처
- hook 이름에 비해 하는 일이 과도하게 많음
- 소비자가 내부 사정을 다 알아야 해서, 추상화가 실제로는 "숨김"이 아니라 "집합"

## 실제 사례

> maxgauge-dashboard — `useAlertLogModalState` hook

### hook 하나에 모든 것을 몰아넣은 구조

```
useAlertLogModalState
  ├─ 선택 상태 (selectedAlert, setSelectedAlert)
  ├─ word wrap 토글
  ├─ 검색 필터
  ├─ 데이터 조회 (API query)
  ├─ pagination reset side effect
  ├─ action builder
  └─ detail-to-viewer 변환
```

이름은 "modal state"인데, 실제로는 query orchestration과 view-data assembly까지 담당한다. 하나의 hook이 7가지 관심사를 가지고 있다.

### 반환 시그니처의 비대함

hook이 상태, setter, handler, 파생 데이터를 거의 전부 노출한다. 소비자는 hook을 쓰면서도 내부 구조를 다 알아야 한다. 이러면 추상화가 아니라 단순 포장이다.

### 과잉 방어 코드

```ts
// AI가 작성한 코드
const resetSelectedAlert = useCallback(() => {
  setSelectedAlert((prev) => prev !== null ? null : prev);
}, []);

// 실제로 필요한 코드
const resetSelectedAlert = useCallback(() => setSelectedAlert(null), []);
```

`prev !== null ? null : prev` — null이 아닐 때만 null로 바꾸는 "조심스러운" 코드. `setSelectedAlert(null)`로 충분하다. AI가 만든 불필요한 방어 로직의 전형적 사례.

## 같은 레포의 또 다른 사례

> [exem-table MR #47](http://gitlab.exem.xyz/fe1/exem-table/-/merge_requests/47) — cell hook 구조

```
Before:
  useCellBehavior (계산값 생성)
  + useCellStateMemo (위 결과를 다시 받아 재계산)
  + useCellAppearance (isRatioColumn으로 분기)

After:
  useCellState (통합)
  + useCellAppearance (variant descriptor로 대체)
```

hook 2개를 "관심사 분리"라는 명목으로 나눴지만, 한 hook의 출력이 다른 hook의 입력이었다. 분리의 이유가 없는 것을 분리한 것.

## 왜 이런 일이 발생하는가

1. **저위험 정리 선호**: 여러 hook을 모아 커스텀 hook으로 감싸는 건 Rules of Hooks를 어기지 않으면서 코드를 "깔끔해 보이게" 만드는 가장 안전한 방법
2. **학습 데이터 편향**: GitHub에 흔한 `useXxxState`, `useXxxActions` 패턴을 프로젝트 맥락 없이 재생산
3. **장황함 편향**: AI는 핸들러 이름을 붙이고, 래퍼를 하나 더 만들고, 구조를 한 단계 더 나누는 쪽을 선호. 연구에 따르면 AI가 만든 코드에 maintainability 문제가 있어도, 평가자 68%는 "가독성이 좋아졌다"고 판단

## 판별 기준

- hook의 **반환값 중 소비자가 안 쓰는 것**이 있으면 시그니처가 비대한 것
- hook을 쓰는 **소비자가 내부 구조를 알아야** 한다면 추상화가 아니라 포장
- hook이 **사용처의 자유도를 줄이지 않는다면** 분리할 이유가 없었던 것
