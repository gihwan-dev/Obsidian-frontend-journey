# 시각적 판단 불가

> AI는 코드에서 색상 이름을 읽을 수 있지만, 그것이 화면에서 어떻게 보이는지는 판단하지 못한다.

시멘틱 토큰이란 색상을 `#ffffff` 같은 직접 값이 아니라 `primary-background` 같은 역할 이름으로 참조하는 방식이다. 같은 토큰이라도 다크/라이트 테마에 따라 실제 색상이 달라진다.

## 현상

- 시멘틱 색상 토큰을 문맥에 맞지 않게 사용
- 대비(contrast) 요구사항을 고려하지 못해 가독성 저하
- 디자인 시스템 마이그레이션 시 기계적으로 토큰만 대응시키고, 시각적 결과를 검증하지 않음

### 문제 1: 시멘틱 토큰 오용

AI가 디자인 시스템을 마이그레이션하면서 tooltip 텍스트에 `text-text-inverse`를 적용했다.

- `text-text-inverse`: 밝은 배경 위 어두운 텍스트용 토큰
- tooltip 배경: 어두운 색

결과적으로 어두운 배경 위에 어두운 텍스트가 올라가 **읽을 수 없는 상태**가 됐다.

### 문제 2: 하드코딩된 fallback 색상

```css
/* AI가 작성한 코드 */
color: var(--color-icon-tertiary, #99a1af);

/* 수정 후 */
color: var(--color-icon-tertiary);
```

AI는 CSS 변수에 기본값(`#99a1af`)을 넣는 "안전한" 패턴을 적용했다. 그러나 디자인 시스템에서는 기본값이 있으면 테마 전환 시 기본값이 우선 적용되어 색상 체계가 무너진다. 토큰을 쓰는 이유 자체가 없어지는 것이다.

## 왜 이런 일이 발생하는가

1. **텍스트 기반 추론의 한계**: AI는 `text-inverse`라는 이름에서 "반전"이라는 의미를 읽지만, 실제로 어떤 색인지, 어떤 배경과 조합되는지를 시각적으로 확인할 수 없다
2. **기계적 매핑**: 토큰 A → 토큰 B로 1:1 대응만 수행하고, 대응 결과가 시각적으로 유효한지 검증하지 않음
3. **맥락 없는 "안전" 패턴**: hex fallback 삽입은 일반적인 CSS 관행이지만, 디자인 시스템 환경에서는 오히려 해로운 패턴

## 교훈

- 색상, 레이아웃, 대비 등 시각적 판단이 필요한 변경은 AI만으로 완결하지 않는다
- 디자인 시스템 마이그레이션에서 AI는 토큰 매핑 초안 생성에는 유용하지만, 최종 검수는 사람이 해야 한다
