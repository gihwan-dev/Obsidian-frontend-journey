# 잘못된 추상화

> AI는 도메인 경계를 발견해서 추상화하는 게 아니라, 학습 데이터에서 자주 본 "그럴듯한 구조"를 재생산한다.

## 현상

- 컴포넌트 이름이 역할을 설명하지 못함
- 추상화가 복잡성을 줄이지 않고 위치만 이동시킴
- pass-through 래퍼(입력을 그대로 하위로 전달하기만 하는 중간 계층)가 쌓여 호출 깊이만 깊어짐

## 실제 사례

> [exem-table MR #47](http://gitlab.exem.xyz/fe1/exem-table/-/merge_requests/47) — feature pipeline 단일화 + cell 경로 응집

### 1. 이름이 역할을 설명하지 못하는 컴포넌트

| AI가 만든 이름 | 실제 역할 | 수정 후 |
|---|---|---|
| `DataCellWrapper` | 유틸리티 + body 조립 | `ValueCell` |
| `DataCellBody` | custom/ratio/normal 분기 | 삭제 (3-way 분기로 대체) |
| `UtilityCellRenderer` | 체크박스/expander 컨트롤 | `ColumnControlCell` |
| `CellContent` | 텍스트/숫자 렌더링 | `TextCell` |

`DataCellWrapper`는 이름만 보면 무슨 역할인지 알 수 없다. 69줄짜리 컴포넌트인데 props 절반이 그대로 하위로 전달되는 pass-through였다.

### 2. 불필요한 숫자 priority 시스템

```ts
// Before: AI가 만든 구조
const FEATURE_DEFINITIONS = [
  { name: 'search', priority: 50, ... },
  { name: 'sort',   priority: 55, ... },
  { name: 'scroll', priority: 95, ... },
  // ... priority 580까지
];
```

```ts
// After: 배열 순서가 곧 우선순위
const FEATURE_DESCRIPTORS = [
  { name: 'search', isEnabled: (opts) => isGlobalSearchEnabled(opts), ... },
  { name: 'sort',   isEnabled: (opts) => !!opts.sort?.use, ... },
  // ...
];
```

숫자 priority는 cargo-cult 복잡성이다. 배열 순서만으로 충분한데 AI는 "더 유연해 보이는" 설계를 선택했다.

### 3. 분리할 이유가 없는 것을 분리

React hook은 상태와 로직을 묶는 함수 단위로, BE의 Service 메서드와 유사하다.

```
Before:
  useCellBehavior → 계산값 생성
  useCellStateMemo → useCellBehavior 결과를 다시 받아 재계산

After:
  useCellState → 단일 hook으로 통합
```

관심사 분리라는 명목으로 함수를 나눴지만, 한쪽의 출력이 다른 쪽의 입력이 되는 직렬 의존이었다. BE로 치면 ServiceA가 계산한 값을 ServiceB가 받아서 다시 가공하는데, 둘 다 같은 트랜잭션 안에서만 호출되는 상황과 같다.

## 왜 이런 일이 발생하는가

1. **학습 데이터의 패턴 재생산**: GitHub에 흔한 `useXxxState`, `useXxxActions` 같은 형태를 프로젝트 맥락과 무관하게 적용
2. **저위험 정리 선호**: 문법 규칙을 어기지 않으면서 코드를 "얇아 보이게" 만드는 가장 안전한 패턴을 선택
3. **추상화 편향**: LLM은 익숙한 코드 패턴을 만나면 실제 분석을 건너뛰고 "기억된 추상화"에 기대는 경향이 있음

## 판별 기준

이 추상화가 **사용처의 자유도를 줄이고 의도를 명확하게 만드는가**, 아니면 **복잡성을 다른 파일로 이동시킨 것뿐인가**를 본다. 전자면 좋은 추상화, 후자면 AI가 만든 "무난한 포장"이다.
