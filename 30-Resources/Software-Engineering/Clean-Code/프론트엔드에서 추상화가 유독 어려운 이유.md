# 프론트엔드에서 추상화가 유독 어려운 이유

> [!quote] 핵심 문장
> 프론트엔드에서 좋은 추상화란 비슷해 보이는 것을 묶는 것이 아니라, 같은 이유로 변하는 것들을 같은 경계 안에 두는 것이다.

## 컴포넌트는 왜 추상화하기 어려운가

백엔드의 비즈니스 로직은 대체로 입력, 출력, 규칙으로 설명할 수 있다. "이게 하나의 문제다"라는 경계가 비교적 선명하다. 함수 하나가 하나의 문제를 풀고, 그 함수를 조합해서 더 큰 문제를 푸는 구조가 자연스럽게 만들어진다.

React 컴포넌트는 다르다. 하나의 컴포넌트 파일 안에 **성격이 전혀 다른 여러 종류의 문제**가 동시에 들어온다.

| 문제의 종류 | 질문 | 예시 |
|---|---|---|
| **마크업/스타일** | 어떻게 보여야 하는가? | JSX 구조, CSS, 레이아웃 |
| **이벤트** | 사용자가 무엇을 할 수 있는가? | 클릭, 입력, 드래그 |
| **상태** | 지금 UI가 어떤 상태에 있는가? | 열림/닫힘, 로딩, 선택됨 |
| **비즈니스 로직** | 어떤 규칙으로 동작해야 하는가? | 유효성 검증, 권한 판단, 가격 계산 |
| **사이드 이펙트** | 외부 세계와 어떻게 상호작용하는가? | API 호출, 라우팅, 로깅, 모달 열기 |

컴포넌트는 단순한 "함수 하나"가 아니다. **시각적 표현 + 상호작용 + 상태 전이 + 도메인 규칙 + 외부 효과가 만나는 접점**이다. 이렇게 이질적인 관심사들이 한 곳에 모여 있으니, "이 컴포넌트의 책임은 무엇인가?"라는 질문에 깔끔하게 답하기가 어렵다. 추상화의 전제 조건인 "명확한 부분 문제 정의"가 처음부터 흔들리는 것이다.

## 마크업 추상화가 특히 위험한 이유

프론트엔드 개발을 하다 보면 "이 JSX 비슷한데, 컴포넌트로 빼볼까?"라는 생각이 자주 든다. 하지만 이 직감은 생각보다 자주 틀린다.

마크업은 **변동성이 매우 큰 층**이기 때문이다.

겉으로 같아 보이는 UI도 실제로는 다를 수 있다:

- 어떤 화면에서는 버튼이 하나 더 필요하다
- 어떤 화면에서는 접근성 속성이 달라진다
- 어떤 화면에서는 레이아웃이 반응형으로 다르게 깨진다
- 어떤 화면에서는 같은 카드처럼 보여도 내부 행동이 완전히 다르다

**겉모양이 비슷하다고 같은 문제는 아니다.** 이것이 비즈니스 로직 추상화와 UI 추상화의 결정적 차이점이다. 비즈니스 로직에서는 "규칙의 공통성"이 비교적 명확하다. 두 함수가 같은 계산을 하고 있다면 실제로 같은 문제를 풀고 있을 확률이 높다. 하지만 UI에서는 **"모양의 공통성"이 실제 공통성이 아닌 경우가 너무 많다.**

JSX가 좀 비슷하다고 성급하게 추상화하면, 시간이 지날수록 요구사항의 미세한 차이를 수용하기 위해 props가 끝없이 늘어나고, 조건 분기가 쌓이고, 결국 원래 컴포넌트보다 더 복잡한 괴물이 탄생한다. "재사용 가능한 컴포넌트"를 만들겠다는 의도가 오히려 유지보수를 어렵게 만드는 아이러니가 발생하는 것이다.

이것은 [[추상화의 본질 - 복잡함을 숨기는 것이 아니라 다룰 수 있는 구조로 바꾸는 것|이전 글]]에서 말한 "분해가 제대로 되지 않은 상태에서 인터페이스만 단순하게 만드는 것은 추상화가 아니라 은폐"라는 원리가 정확히 적용되는 사례다.

## 진짜 추상화 대상은 컴포넌트가 아니라 "반복되는 문제"다

여기서 중요한 전환이 필요하다. **추상화 대상이 항상 컴포넌트일 필요는 없다.**

프론트엔드에서 "반복"이 일어나는 곳을 자세히 들여다보면, 반복되는 것의 성격에 따라 추상화 방식이 완전히 달라져야 한다.

### 1. 시각적 문제의 반복

같은 모양, 같은 시맨틱, 같은 디자인 규칙이 반복될 때. 이 경우에만 UI 컴포넌트로 추상화하는 것이 자연스럽다. 단, 여기서 "같은 모양"은 "우연히 비슷해 보이는 것"이 아니라 **"의도적으로 같아야 하는 것"**을 의미한다. 디자인 시스템에서 정의한 Button, Input, Card, Modal 같은 것들이 여기에 해당한다. 이것들은 "이 모양이 바뀌면 모든 곳에서 함께 바뀌어야 한다"는 의도가 있기 때문에 추상화가 정당하다.

### 2. 상호작용의 반복

같은 상태 전이, 같은 이벤트 처리 흐름이 반복될 때. 열기/닫기, 선택/해제, 폼 입력/검증/제출 같은 패턴들이 여기에 해당한다. 이것들은 **모양과 무관하게 행동이 동일**하다. 아코디언이든 드롭다운이든 모달이든, "열고 닫는다"는 행동 구조는 같다. 이런 경우 커스텀 훅, headless component, 또는 state machine으로 추상화하는 것이 적절하다. 시각적 표현은 밖에 두고 행동만 캡슐화하는 것이다.

### 3. 도메인 규칙의 반복

같은 비즈니스 규칙, validation, 권한 판단, 데이터 변환 로직이 반복될 때. 가격 계산, 주문 가능 여부 판단, 폼 검증 규칙 같은 것들이다. 이것들은 UI와 완전히 독립적이다. 어떤 화면에서 쓰이든, 어떤 컴포넌트에 들어가든 규칙 자체는 변하지 않는다. 서비스 함수, use-case, 도메인 훅으로 추상화해서 UI 계층과 분리해야 한다.

### 4. 외부 효과의 반복

같은 데이터 요청, 저장, 라우팅, 토스트 알림, 이벤트 로깅이 반복될 때. 이것들은 "외부 세계와의 접점"이라는 공통성을 가진다. query/mutation 훅이나 effect orchestration 레이어로 추상화한다.

> 프론트엔드 추상화의 출발점은 "컴포넌트를 어떻게 뽑을까?"가 아니라, **"지금 반복되는 것이 모양인가, 행동인가, 규칙인가, 외부 효과인가?"**라는 질문이어야 한다.

## 상태가 모든 것을 어렵게 만드는 이유

위에서 컴포넌트 안의 여러 관심사를 나열했지만, 이것들을 깔끔하게 분리하기 어려운 근본적인 원인이 하나 있다. **상태가 모든 층을 접착제처럼 붙잡고 있기 때문**이다.

상태는 동시에 여러 곳에 발을 걸치고 있다:
- UI와 붙어 있다 (열림/닫힘이 렌더링을 결정한다)
- 이벤트와 붙어 있다 (클릭이 상태를 바꾼다)
- 비즈니스 로직과 붙어 있다 (상태에 따라 다른 규칙이 적용된다)
- 서버 데이터와 붙어 있다 (API 응답이 상태를 채운다)

그래서 상태의 경계를 잘못 잡으면 전체 추상화 구조가 무너진다. 하나의 상태 객체가 너무 많은 관심사를 품고 있으면, 그 상태를 사용하는 모든 코드가 서로 암묵적으로 결합된다.

해결의 실마리는 **상태의 성격을 구분하는 것**에 있다:

| 상태의 성격 | 예시 | 변하는 이유 |
|---|---|---|
| **시각적 상태** | 열림/닫힘, hover, 탭 선택 | 사용자 인터랙션 |
| **입력 상태** | 폼 값, 체크 여부 | 사용자 입력 |
| **서버 상태** | loading, error, data | 네트워크 응답 |
| **도메인 상태** | 현재 주문 단계, 권한, 완료 여부 | 비즈니스 규칙 |

이것들을 한 컴포넌트 안에서 한 덩어리로 관리하면 복잡해진다. 반대로, 각 상태를 성격에 맞는 추상화 단위로 분리하면 — 시각적 상태는 UI 훅으로, 서버 상태는 query 훅으로, 도메인 상태는 비즈니스 로직으로 — 각각이 독립적으로 이해 가능하고 테스트 가능해진다.

## 그러면 문제를 어떻게 정의해야 하나

프론트엔드에서의 문제 분해는 **두 단계**로 이루어져야 한다.

### 1단계: 시각적 구조로 1차 분해

화면을 보고 직관적으로 시각적 단위를 나눈다. 헤더, 리스트, 필터 패널, 상세 카드, 폼 영역. 이것은 우리 눈에 자연스러운 분해 방식이고, 초기 구조를 잡는 데 좋은 출발점이다.

### 2단계: 변화의 이유로 2차 분해

각 시각적 단위 안에서 다시 질문한다:

- 여기서 **자주 바뀌는 건 모양**인가? → 안정적이면 UI 컴포넌트로 고정
- **행동 흐름**이 반복되는가? → 커스텀 훅으로 분리
- **도메인 규칙**이 반복되는가? → 비즈니스 로직으로 분리
- **외부 효과**가 반복되는가? → query/mutation 계층으로 분리

시각적 구조는 1차 분해 기준으로서 유용하지만, **재사용 가능한 추상화의 경계는 "같이 변하는 것들"을 기준으로 잡아야 한다.** 시각적으로 하나인 것이 추상화에서도 하나라는 보장은 없다. 반대로, 시각적으로 떨어져 있어도 같은 이유로 변하는 것들은 같은 추상화 안에 있어야 할 수 있다.

## Render, Behavior, Orchestration — 세 가지 층

이 모든 논의를 관통하는 실용적인 프레임워크가 있다.

### Render — 어떻게 보이느냐

순수 UI 컴포넌트. props를 받아서 렌더링한다. 사이드 이펙트 없이, 상태는 최소한으로. 디자인 시스템과 표현 책임에 집중한다. "같은 props를 주면 항상 같은 결과가 나온다"는 예측 가능성이 핵심이다.

### Behavior — 어떻게 반응하느냐

상호작용 추상화. 이벤트 핸들러, 로컬 상태, 파생 상태, 복잡한 UI 행동 흐름을 담는다. 커스텀 훅이 대표적인 구현 수단이다. 시각적 표현과 분리되어 있기 때문에, 같은 행동을 서로 다른 UI에 적용할 수 있다.

### Orchestration — 외부 세계와 어떻게 연결되느냐

페이지/피처 조립 레이어. API 호출 연결, 라우팅, 여러 훅과 UI의 조합, 실제 화면 단위의 조율을 담당한다. 이 층은 재사용을 목적으로 하지 않는다. 특정 화면의 특정 시나리오를 해결하는 것이 목적이다.

이 세 층의 구분이 명확할수록 각 층은 독립적으로 변경 가능해지고, 한 층의 변경이 다른 층에 미치는 영향을 최소화할 수 있다.

## 핵심 원칙: 비슷해 보이는 것이 아니라 같은 이유로 변하는 것을 묶어라

프론트엔드에서 추상화 판단을 할 때 가장 중요한 기준은 이것이다:

> **겉모양이 비슷해도 바뀌는 이유가 다르면 분리해야 하고, 겉모양이 달라도 행동이나 규칙이 같으면 같은 추상화로 묶을 수 있다.**

이것은 결국 "변경의 축(axis of change)"을 기준으로 경계를 긋는다는 뜻이다. 같은 이유로 함께 변하는 것들은 같은 경계 안에, 다른 이유로 변하는 것들은 다른 경계에 두어야 한다. 이 원칙을 무시하고 시각적 유사성만으로 추상화하면, 시간이 지날수록 요구사항의 분기가 추상화의 내부를 복잡하게 만들고, 결국 추상화를 깨뜨리거나 더 큰 복잡성을 만들어낸다.

## 글에 쓸 만한 핵심 문장들

- 컴포넌트는 함수 하나가 아니라, 시각적 표현과 상호작용과 상태 전이와 도메인 규칙과 외부 효과가 만나는 접점이다.
- UI에서 "모양의 공통성"은 실제 공통성이 아닌 경우가 너무 많다.
- 프론트엔드 추상화의 출발점은 "컴포넌트를 어떻게 뽑을까?"가 아니라 "반복되는 것의 성격이 무엇인가?"다.
- 상태는 UI, 이벤트, 로직, 서버 데이터를 접착제처럼 붙잡고 있어서, 상태 경계를 잘못 잡으면 전체 구조가 무너진다.
- 시각적 구조로 먼저 나누고, 변화의 이유로 다시 나눠라.
- 프론트엔드에서 좋은 추상화란 비슷해 보이는 것을 묶는 것이 아니라, 같은 이유로 변하는 것들을 같은 경계 안에 두는 것이다.
