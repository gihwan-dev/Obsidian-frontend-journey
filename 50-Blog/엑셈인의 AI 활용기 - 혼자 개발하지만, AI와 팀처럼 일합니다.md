# 엑셈인의 AI 활용기 : 혼자 개발하지만, 이제는 AI와 팀처럼 일합니다

**자동완성 도구에서 시작한 AI 활용은, 이제 역할을 나눠 함께 일하는 개발 방식으로 바뀌었습니다. 프론트엔드 개발자로서 제가 AI를 '도구'가 아니라 '팀원'처럼 쓰게 된 과정을 정리했습니다.**

---

## 동시에 MR 3개를 올린 날

얼마 전, 기능 추가 작업 세 건을 동시에 진행해서 MR(코드 반영 요청) 3개를 동시에 올린 적이 있습니다. 예전 같았으면 반나절은 걸렸을 작업입니다. 하나 끝내고, 다음 거 시작하고, 또 끝내고. 그렇게 순서대로 했을 테니까요.

그날은 달랐습니다. 세 작업이 동시에 돌아가고 있었고, 저는 각각의 진행 상황을 확인하면서 방향을 잡아주는 역할만 했습니다.

이게 어떻게 가능했는지 설명하려면, 두 가지를 먼저 알아야 합니다.

---

## 먼저 알면 좋은 것 ①: 워크트리

개발자가 하나의 프로젝트에서 작업할 때, 보통은 **한 번에 하나의 작업만** 진행합니다. 기능 A를 만들다가 기능 B를 시작하려면, A를 일단 저장해두고 전환해야 합니다. 마치 보고서를 쓰다가 다른 보고서를 쓰려면 파일을 닫고 새로 열어야 하는 것과 비슷합니다.

**워크트리(Worktree)**는 이 제약을 없애줍니다. 같은 프로젝트를 여러 개의 독립된 작업 공간으로 복사해서, 각각 다른 작업을 동시에 진행할 수 있게 해줍니다.

쉽게 비유하면 이렇습니다. 하나의 보고서 원본이 있고, \이걸 세 개의 사본으로 나눕니다. **1번 사본에서는 서론을 고치고, 2번에서는 표를 추가하고, 3번에서는 결론을 다듬습니다.** 세 작업이 서로 방해하지 않고, 다 끝나면 원본에 하나씩 합칩니다.

제가 MR 3개를 동시에 올릴 수 있었던 건, 바로 이 워크트리 덕분입니다. 하나의 프로젝트에서 세 개의 작업 공간을 열고, 각각에 AI를 붙여서 동시에 돌렸습니다.

---

## 먼저 알면 좋은 것 ②: 멀티 에이전트

그런데 워크트리만으로는 부족합니다. 작업 공간이 세 개여도, **일할 사람이 나 하나면** 결국 돌아가면서 해야 하니까요.

여기서 **멀티 에이전트**가 등장합니다.

요즘 AI 도구는 하나의 AI가 모든 걸 하는 게 아니라, **역할이 다른 여러 AI가 팀처럼 움직이는** 구조로 발전하고 있습니다. 제가 쓰는 환경을 예로 들면 이렇습니다.

- **탐색 담당**: 기존 코드를 읽고 "이 기능은 여기를 고쳐야 한다"를 파악합니다
- **구현 담당**: 실제 코드를 작성합니다
- **검토 담당**: 작성된 코드에 문제가 없는지 자동으로 점검합니다
- **구조 분석 담당**: 코드가 너무 복잡해지지 않았는지 확인합니다

비유하자면, **제가 팀장이고 AI가 팀원 여럿인 조직**과 비슷합니다. 제가 "이거 만들어줘"라고 하면, AI 팀장(오케스트레이터)이 알아서 탐색팀에게 조사 시키고, 구현팀에게 만들라 하고, 검토팀에게 확인시킵니다. 저한테는 최종 결과만 올라옵니다.

---

## 근데 처음부터 이랬던 건 아닙니다

올해 초, Cursor를 처음 쓰기 시작했을 때는 이야기가 달랐습니다.

AI가 코드를 대신 써주긴 했지만, 지시를 아주 세세하게 내려야 했습니다. "이 이름으로, 이런 입력을 받아서, 이런 동작을 하는 화면 조각을 만들어줘"라고 구조적으로 써야 원하는 결과가 나왔거든요. 비유하자면 **아주 꼼꼼한 지시서를 써야만 움직이는 신입 사원** 같았습니다.

무엇보다 **한 번에 하나의 작업만** 가능했습니다. 워크트리도 없고, 멀티 에이전트도 없으니 당연합니다. 기능 하나를 처음부터 끝까지 내가 직접 이끌어야 했고, 결과물도 한 줄 한 줄 직접 검증해야 했습니다.

솔직히 이 시기의 AI는 개발이 편해진 정도였지, 일하는 방식 자체가 바뀌었다는 느낌은 아니었습니다.

---

## 전환점: AI가 계획을 세우고, 스스로 고치기 시작했습니다

작년 6월쯤 Claude Code를 쓰기 시작했고, 여기서부터 흐름이 바뀌었습니다.

가장 큰 변화는 **계획 모드(Plan Mode)**의 등장이었습니다. AI가 바로 코드를 쓰는 게 아니라, "이렇게 만들려고 하는데 괜찮을까요?"라고 **먼저 계획을 보여주는** 겁니다. 저는 그 계획을 보고 방향을 조정한 뒤 "진행해"라고 승인합니다. 마치 팀원이 작업 전에 설계안을 먼저 공유하는 것과 같습니다.

그리고 서브 에이전트가 생기면서, 앞서 설명한 멀티 에이전트 구조가 가능해졌습니다. 이전에는 제가 직접 파일 위치를 알려주면서 "여기를 이렇게 고쳐"라고 해야 했는데, 이제는 AI가 알아서 코드를 탐색하고, 어디를 고칠지 판단하고, 구현하고, 검토까지 합니다.

특히 **"검토 → 수정" 사이클을 AI가 스스로 반복하는** 구조가 만들어졌습니다. 예전에는 코드를 만들면 제가 직접 읽고, 문제를 찾고, 고치라고 지시해야 했습니다. 지금은 AI가 코드를 쓰고 → 자동으로 검토하고 → 문제가 있으면 스스로 고치고 → 다시 검토하는 과정을 알아서 반복합니다.

저는 이 사이클이 끝난 결과만 확인합니다. 화면이 의도대로 동작하는지 보고, 제가 정한 품질 기준에 맞는지 확인합니다. 문제가 있으면 피드백을 주고, AI는 다시 검토-수정 사이클을 돌립니다. 문제가 없으면 완료.

비유하자면, 예전에는 제가 **직접 글을 쓰고, 직접 퇴고하고, 직접 교정까지 하는 1인 작가**였다면, 지금은 **방향을 잡아주고 최종 원고만 확인하는 편집장**에 가깝습니다.

---

## 다시 그날 이야기로 돌아오면

이제 MR 3개를 동시에 올린 날의 그림이 그려지실 겁니다.

저는 세 개의 워크트리를 열고, 각 작업 공간에 AI를 하나씩 붙였습니다. 각 AI는 자기 작업 공간에서 코드를 탐색하고, 계획을 세우고, 구현하고, 스스로 검토하고, 문제가 있으면 수정하는 사이클을 돌렸습니다.

저는 그 사이를 오가며 계획을 승인하고, 결과를 확인하고, 방향을 조정했습니다. **세 개의 작업이 동시에 진행되는 동안, 제가 한 건 설계와 판단과 검수였습니다.** 코드를 직접 쓴 건 거의 없었습니다.

예전에는 반나절이 걸렸을 일이 2시간 만에 끝났습니다.

---

## 그래서 지금 개발자의 역할은 뭘까요

AI가 들어오면서 줄어든 건 직접 코드를 쓰고 검토하는 수고였습니다. 하지만 더 중요해진 건 따로 있었습니다.

작업을 어떤 단위로 나눌지 정하는 일. AI의 계획이 올바른 방향인지 판단하는 일. 자동 검토를 통과한 결과물이 실제 제품 맥락에 맞는지 확인하는 일. 품질 기준과 우선순위를 세우는 일. 이런 것들이 제 하루의 대부분을 차지하게 됐습니다.

**AI가 바꾼 건 코딩 속도보다, 일을 설계하고 판단하고 검수하는 방식이었습니다.**

그리고 이건 개발자만의 이야기가 아닙니다. AI는 "다 해줘"라고 할 때보다, 단계를 나눠줄수록 더 잘 작동합니다. 기획서를 쓸 때도 "전체 기획서 써줘"보다 "이 세 가지 관점으로 각각 초안을 뽑아줘"라고 하면 결과가 달라집니다. 결과물을 바로 쓰기보다, **계획을 먼저 확인하고, 결과를 검수하는 단계**를 넣을수록 AI를 더 잘 쓸 수 있습니다.

결국 중요한 건 더 좋은 답변을 받는 기술이 아니라, **일을 잘 나누고, 계획을 확인하고, 결과를 판단하는 방식**입니다.
