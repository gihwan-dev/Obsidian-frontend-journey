자동완성 도구에서 시작한 AI 활용은, 이제 여러 작업을 나눠 동시에 진행하는 방식으로 확장되고 있습니다. 이 글에서는 프론트엔드 개발자로서 제가 최근 AI를 어떻게 활용하고 있는지, 생산성 향상의 두 가지 축을 이루는 워크트리와 멀티 에이전트를 중심으로 소개해보려 합니다.

## 요즘 AI로 개발하는 방식

AI 코딩 도구를 처음 쓸 때는 주로 자동완성, 코드 초안 작성, 반복 작업 보조처럼 개발 과정의 일부를 돕는 용도로 활용했습니다. 최근에는 활용 방식이 조금 달라졌습니다. 동시에 여러 작업을 수행하며 계획과 설계, 그리고 결과물을 검증하는 형태로 일하고 있습니다. 코드 타이핑은 전혀 하지 않고요.

![[스크린샷 2026-03-09 오후 5.36.42.png]]

그 변화의 중심에는 두 가지가 있습니다. 하나는 Git의 워크트리를 활용해 여러 작업을 병렬로 진행하는 방식이고, 다른 하나는 탐색, 구현, 리뷰를 나눠 수행하는 멀티 에이전트 구조입니다. 아래에서는 이 두 가지를 중심으로, 제가 요즘 어떤 방식으로 개발하고 있는지 소개해보겠습니다.

## 워크트리로 여러개 프로젝트 동시에 작업하기

개발자는 코드를 관리하기 위해 **Git**이라는 도구를 씁니다. 쉽게 말하면, 코드를 변경할 때마다 **그 시점의 스냅샷을 찍어두는** 도구입니다. 언제든 과거 시점으로 돌아갈 수 있고, "이 시점에서 갈래를 나눠서 다른 작업을 해보자"는 것도 가능합니다.

![[Pasted image 20260317142705.png]]

이 Git에는 **워크트리(Worktree)**라는 기능이 있습니다. 같은 프로젝트를 **여러 개의 독립된 작업 공간으로 복제**해서, 각 공간에서 서로 다른 작업을 동시에 진행할 수 있게 해주는 기능입니다.

![[Pasted image 20260318081128.png]]

비유하자면 이렇습니다. 원래는 **책상 하나에서 서류를 한 건씩 처리하는 방식**이었다면, 워크트리는 **책상을 여러 개 펼쳐놓고 각 책상에 복사된 서류를 올려놓는 것**입니다. 각 책상에서 필요한 작업을 동시에 진행할 수 있는거죠.

![[Pasted image 20260318075913.png]]

그런데 이 기능은 오랫동안 거의 쓰이지 않았습니다. 이유는 단순합니다. **사람은 어차피 동시에 두 가지를 코딩할 수 없으니까요.** 책상을 세 개 펼쳐놔도 일하는 사람이 나 하나면, 결국 돌아가면서 해야 합니다. 워크트리는 있으나 마나 한 기능이었습니다.

![[Pasted image 20260318080846.png]]

**AI가 나오면서 이야기가 완전히 달라졌습니다.**

각 책상에 AI 직원을 한 명씩 배치할 수 있게 된 겁니다. 워크트리 3개를 만들고, 각각에 AI를 붙이면, **세 개의 작업이 동시에 진행됩니다.** 저는 그 사이를 오가며 방향만 잡아주면 됩니다.

![[Pasted image 20260318081229.png]]

Claude Desktop, Codex Desktop, Conductor, OpenCode 등 다양한 도구에서 이 워크트리 기능을 간편하게 사용할 수 있는 UI를 제공하고 있습니다. 그덕에 최근에는 터미널이 아닌 데스크톱 앱에서 편하게 병렬작업을 수행할 수 있게 되었습니다.

![[스크린샷 2026-03-18 오전 8.22.44.png]]

## 멀티 에이전트 기반 자가 피드백 루프로 코드 품질 개선하기

병렬로 돌리는 건 좋은데, 한 가지 문제가 남습니다. **AI가 만든 코드의 품질은 괜찮은가?** 코드 품질이 개선되지 않는다면, 이를 리뷰 하는 시간이 생산성에 병목이 될거라 생각했습니다.

저도 오랫동안 그랬습니다. 그런데 경험이 쌓이면서, AI의 결과물이 안 좋을 때의 원인이 크게 두 가지로 좁혀졌습니다.

**첫 번째, 불필요한 정보로 컨텍스트가 오염되는 경우.** AI에게 작업을 시킬 때, 관련 없는 정보가 많이 섞여 들어가면 AI가 초점을 잃습니다. 사람도 회의 중에 주제와 관계없는 이야기가 길어지면 집중력이 떨어지는 것과 비슷합니다.
![[Pasted image 20260318083214.png]]

**두 번째, 피드백 루프가 부족한 경우.** AI가 코드를 쓰고 바로 끝이면, 당연히 품질이 들쭉날쭉합니다. 사람이 글을 쓸 때도 초고를 바로 제출하면 품질이 안 나오는 것과 같은 이치입니다.
![[Pasted image 20260318083310.png]]

이 두 문제를 **멀티 에이전트 구조**로 해결했습니다.

멀티 에이전트란, 하나의 AI가 모든 걸 하는 게 아니라 **역할이 다른 여러 AI가 팀처럼 움직이는 구조**입니다.

- **탐색 에이전트**: 코드를 읽고 필요한 정보만 골라서 가져옵니다. 불필요한 정보가 섞이지 않도록 컨텍스트를 깔끔하게 유지하는 역할입니다.
- **구현 에이전트**: 탐색 결과를 바탕으로 코드를 작성합니다.
- **리뷰 에이전트**: 작성된 코드를 자동으로 검토합니다. 코드 품질, 구조적 복잡도, 패턴 위반 등을 점검합니다.
- **관리 에이전트**: 작업 지시를 바탕으로 어떤 AI를 소환해서 어떤 작업을 수행시킬지 이 모든 흐름을 관리합니다.

핵심은 **리뷰 에이전트가 문제를 발견하면, 구현 에이전트가 수정하고, 다시 리뷰가 돌아가는 사이클을 AI끼리 자동으로 반복**한다는 점입니다. 예전에는 제가 직접 코드를 읽고 "여기 고쳐"라고 했던 작업을, 이제는 AI가 스스로 돌립니다.

## 이건 저만의 이야기가 아닙니다

## 이건 저만의 이야기가 아닙니다

최근 AI 개발 도구들은 공통적으로 비슷한 방향으로 발전하고 있습니다.

- **검증-수정 루프**: AI가 초안을 만든 뒤 테스트, 린트, 리뷰 결과를 확인하고 다시 고치는 반복 실행 구조입니다.
- **멀티 에이전트**: 하나의 AI에게 모든 일을 맡기기보다, 탐색·구현·리뷰처럼 역할을 나눠 여러 AI가 협업하는 구조입니다.
- **워크트리 기반 병렬 작업**: 같은 프로젝트를 서로 간섭하지 않는 작업 공간으로 나눠 여러 작업을 동시에 진행하는 방식입니다.

최근에는 이런 접근을 **'하네스 엔지니어링'**이라고 부르기도 합니다. 프롬프트를 잘 쓰는 것만으로는 부족하고, 문서, 규칙, 테스트, 관측 정보, 피드백 루프까지 포함해 **AI가 안정적으로 일할 수 있는 환경 자체를 설계하는 일**이 점점 중요해지고 있다는 뜻입니다.

## 그래서 지금 저의 역할은

지금 제 하루의 대부분은 **계획 검토와 최종 검수**입니다. AI가 세운 계획이 방향에 맞는지 확인하고, AI가 스스로 검토-수정을 거친 결과물을 최종 확인합니다.

그리고 나머지 시간은 **AI 환경 자체를 개선하는 데** 씁니다. 결과물에서 부족한 부분이 반복적으로 보이면, "이건 AI가 매번 놓치는 패턴이네"라고 파악하고, 그 부분을 자동으로 잡아주는 리뷰 규칙을 추가합니다. AI가 일하는 환경을 점점 더 정교하게 다듬어나가는 거죠.

코드를 직접 쓰는 시간은 확실히 줄었습니다. 대신 **어떻게 나눠서 맡기고, 어떤 기준으로 검수하고, 어떤 환경에서 AI가 더 잘 작동하는지**를 고민하는 시간이 늘었습니다.

## 마무리  
  
요즘 저는 AI를 '대신 코드를 짜주는 도구'보다는, 여러 작업을 나눠 맡길 수 있는 협업 도구에 가깝게 사용하고 있습니다. 특히 워크트리와 멀티 에이전트 방식은, 한 번에 하나씩 처리하던 개발을 여러 갈래로 나눠 동시에 진행할 수 있게 해줬다는 점에서 체감이 컸습니다.  

아직 정답이 정해진 방식은 아니고, 도구도 빠르게 바뀌고 있습니다. 다만 분명한 건, AI에게 얼마나 길게 설명하느냐보다 어떤 구조에서 일하게 하느냐가 점점 더 중요해지고 있다는 점입니다. 이 글이 “요즘 개발자는 AI를 이렇게 활용하기도 하는구나” 정도의 감각을 전하는 데 도움이 되었으면 합니다.
