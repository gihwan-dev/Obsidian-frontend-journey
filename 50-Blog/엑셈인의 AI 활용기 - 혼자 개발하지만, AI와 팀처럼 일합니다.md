# 엑셈인의 AI 활용기 : 혼자 개발하지만, 이제는 AI와 팀처럼 일합니다

**자동완성 도구에서 시작한 AI 활용은, 이제 역할을 나눠 함께 일하는 개발 방식으로 바뀌었습니다. 프론트엔드 개발자로서 제가 AI를 '도구'가 아니라 '팀원'처럼 쓰게 된 과정을 정리했습니다.**

---

## 어제 기능 하나를 만들면서 AI에게 이렇게 지시했습니다

"이 화면에 필터 기능 추가해줘." 한 줄이 전부였습니다.

그러자 AI가 알아서 기존 코드를 읽고, 어디를 고쳐야 할지 파악하고, 계획을 세워서 보여줬습니다. 제가 "좋아, 진행해"라고 하자 구현을 시작했고, 완성된 코드를 스스로 검토한 뒤 문제가 있으면 고치고, 다시 검토하고, 또 고치는 과정을 반복했습니다. 저는 최종 결과만 확인하고 "이 부분만 수정해줘"라고 한 번 더 피드백했을 뿐입니다.

1년 전의 저에게 이 장면을 보여줬다면 믿지 않았을 겁니다. AI는 빠른 자동완성 정도로 생각했으니까요. 그런데 지금 돌아보면, 바뀐 건 AI의 성능보다 **제가 일을 다루는 방식**이었습니다.

---

## 처음엔 빠른 보조 도구였습니다

올해 초, Cursor를 처음 쓰기 시작했을 때는 꽤 혁신적이었습니다. 복잡한 알고리즘 구현을 맡기거나, 테스트 코드와 스토리북 작성을 시킬 수 있었으니까요.

다만 딱 거기까지였습니다. 코드 구현은 제가 직접 하거나, 결과물을 한 줄 한 줄 검증해야 했습니다. 무엇보다 지시를 아주 자세하게 내려야 했습니다. "이 이름으로, 이런 입력을 받아서, 이런 동작을 하는 화면 조각을 만들어줘"라고 구조적으로 써야 원하는 결과가 나왔습니다.

비유하자면 **아주 꼼꼼한 지시서를 써야만 움직이는 신입 사원** 같았습니다. 일을 시킬 수는 있지만, 지시서 쓰는 데 드는 시간을 생각하면 생산성이 올랐다는 느낌은 아니었습니다.

빨라지긴 했지만, **일하는 방식 자체가 바뀌진 않았습니다.**

---

## 그다음엔 실행하는 동료가 되었습니다

작년 6월쯤 Claude Code를 쓰기 시작했고, 이게 전환점이었습니다.

가장 큰 변화는 지시 방식이었습니다. 이전처럼 세세한 지시서를 쓰지 않아도 됐습니다. "이 화면에서 로딩이 너무 느린데, 원인 찾아서 고쳐줘"처럼 자연스럽게 말하면 AI가 스스로 코드를 읽고, 원인을 파악하고, 수정 방향을 잡았습니다.

여기서 중요한 기능이 하나 등장합니다. **계획 모드(Plan Mode)**입니다. AI가 바로 코드를 고치는 게 아니라, "이렇게 고치려고 하는데 괜찮을까요?"라고 먼저 계획을 보여주는 거죠. 저는 그 계획을 보고 방향을 조정한 뒤 "진행해"라고 승인합니다. 마치 **팀원이 작업 전에 설계안을 먼저 공유하는 것**과 같습니다.

이때부터 제가 신경 쓰는 포인트도 바뀌었습니다. 예전에는 "지시를 얼마나 정교하게 쓰느냐"가 중요했다면, 이제는 **"필요한 맥락을 얼마나 잘 전달하느냐"**가 핵심이 됐습니다. 비개발자 분들께 비유하자면, 예전의 AI가 "잘 아는 검색창"이었다면, 이때부터는 **"업무 배경을 이해하고 직접 일까지 해보는 조수"**에 가까워진 겁니다.

---

## 진짜 변화는 AI가 스스로 검토하고 고치기 시작하면서 왔습니다

최근 가장 큰 전환점은 **AI가 자기 결과물을 스스로 점검하는 구조**가 만들어진 겁니다.

지금 제 작업 흐름은 이렇습니다.

**1단계: 작업 지시.** 저는 "이 화면에 필터 기능 추가해줘"처럼 자연어로 지시합니다.

**2단계: AI가 조사하고 계획을 세웁니다.** AI는 혼자 일하지 않습니다. 내부적으로 여러 역할의 보조 AI를 거느리고 있어요. 코드를 탐색하는 AI, 외부 문서를 조사하는 AI, 코드 구조를 분석하는 AI가 각각 필요한 정보를 모아옵니다. 비유하자면 **팀장이 팀원들에게 조사를 시키는 것**과 비슷합니다. 이 결과를 종합해서 "이렇게 만들겠습니다"라는 계획을 내놓습니다.

**3단계: 제가 계획을 검토합니다.** 이 과정이 중요합니다. AI가 세운 계획이 제 의도와 맞는지 확인하고, 방향이 다르면 피드백합니다. AI는 피드백을 반영해서 계획을 다시 세웁니다.

**4단계: 구현합니다.** 승인된 계획대로 AI가 코드를 작성합니다.

**5단계: AI가 스스로 검토하고 수정합니다.** 여기가 가장 큰 변화입니다. 구현이 끝나면 자동으로 코드 품질 검토가 돌아갑니다. 문제가 발견되면 AI가 스스로 수정합니다. 수정이 끝나면 다시 검토합니다. **이 "검토 → 수정" 사이클을 AI가 알아서 반복합니다.** 예전에는 제가 직접 하던 일입니다.

**6단계: 제가 최종 검수합니다.** AI의 자체 검토가 끝나면 저는 최종 결과를 확인합니다. 화면이 의도대로 동작하는지 보고, 제가 정한 품질 기준에 맞는지 확인합니다. 문제가 있으면 피드백을 주고, AI는 다시 검토-수정 사이클을 돌립니다. 문제가 없으면 작업이 완료됩니다.

비유하자면, 예전에는 제가 **직접 글을 쓰고, 직접 퇴고하고, 직접 교정까지 하는 1인 작가**였다면, 지금은 **방향을 잡아주고 최종 원고만 확인하는 편집장**에 가깝습니다. 초고 작성도, 퇴고도, 교정도 AI가 스스로 순환하며 처리합니다.

**바뀐 건 도구가 아니라, 일의 순환 구조 자체였습니다.**

---

## 그래서 지금 개발자의 역할은 뭘까요

AI가 들어오면서 줄어든 건 직접 코드를 쓰고 검토하는 수고였습니다. 하지만 더 중요해진 건 따로 있었습니다.

작업을 어떤 단위로 나눌지 정하는 일. AI의 계획이 올바른 방향인지 판단하는 일. 자동 검토를 통과한 결과물이 실제 제품 맥락에 맞는지 확인하는 일. 품질 기준과 우선순위를 세우는 일. 이런 것들이 제 하루의 대부분을 차지하게 됐습니다.

**AI가 바꾼 건 코딩 속도보다, 일을 설계하고 판단하고 검수하는 방식이었습니다.**

그리고 이건 개발자만의 이야기가 아닙니다. AI는 "다 해줘"라고 할 때보다, 단계를 나눠줄수록 더 잘 작동합니다. 기획서를 쓸 때도 "전체 기획서 써줘"보다 "이 세 가지 관점으로 각각 초안을 뽑아줘"라고 하면 결과가 달라집니다. 결과물을 바로 쓰기보다, 계획을 먼저 확인하고, 결과를 검수하는 단계를 넣을수록 AI를 더 잘 쓸 수 있습니다.

결국 중요한 건 더 좋은 답변을 받는 기술이 아니라, **문제를 잘 나누고, 계획을 확인하고, 결과를 판단하는 방식**입니다.
