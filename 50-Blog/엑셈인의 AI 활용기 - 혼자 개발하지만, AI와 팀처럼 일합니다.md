요즘 저는 하나의 프로젝트를 여러 작업 공간으로 나눠 두고, 각 공간에서 AI에게 서로 다른 일을 맡깁니다. 

- 오류 수정
- 코드 구조 정리
- 테스트 보강
- etc...

예전에는 자동완성이나 코드 초안 작성처럼 개발 과정의 일부를 돕는 용도로만 AI를 썼다면, 요즘은 여러 작업을 병렬로 진행하고 그 결과를 검증하는 흐름 자체를 설계하는 데 더 가깝습니다.

![[동시 여러개 작업.png]]

이 변화의 중심에는 두 가지가 있습니다. 하나는 워크트리로 처리량을 늘리는 방식이고, 다른 하나는 멀티 에이전트로 품질을 안정화하는 방식입니다. 이 글에서는 프론트엔드 개발자인 제가 최근 AI를 어떻게 활용하고 있는지, 이 두 축을 중심으로 소개해보려 합니다.

## 한 프로젝트를 여러 작업 공간으로 나눠 일하기

개발자는 코드를 관리하기 위해 **Git**이라는 도구를 씁니다. 쉽게 말하면, 코드를 변경할 때마다 **그 시점의 스냅샷을 찍어두는** 도구입니다. 언제든 과거 시점으로 돌아갈 수 있고, “이 시점에서 갈래를 나눠 다른 작업을 해보자”는 것도 가능합니다.

![[Pasted image 20260317142705.png]]

이 Git에는 **워크트리(Worktree)**라는 기능이 있습니다. 같은 프로젝트를 여러 개의 독립된 작업 공간처럼 펼쳐 두고, 각 공간에서 서로 다른 작업을 진행할 수 있게 해주는 기능입니다.

![[Pasted image 20260318081128.png]]

비유하자면 이렇습니다. 원래는 **책상 하나에서 서류를 한 건씩 처리하는 방식**이었다면, 워크트리는 **책상을 여러 개 펼쳐놓고 각 책상에 복사된 서류를 올려놓는 것**에 가깝습니다. 각 책상에서 필요한 작업을 동시에 진행할 수 있는 거죠.

![[Pasted image 20260318075913.png]]

그런데 이 기능은 오랫동안 크게 주목받지 못했습니다. 이유는 단순합니다. 사람은 어차피 두 가지 작업을 동시에 제대로 하기 어렵기 때문입니다. 책상을 세 개 펼쳐놔도, 일하는 사람이 나 하나라면 결국 돌아가면서 해야 합니다.

![[Pasted image 20260318080846.png]]

그런데 AI가 등장하면서 이야기가 완전히 달라졌습니다.

각 책상에 AI를 한 명씩 붙일 수 있게 된 겁니다. 워크트리 3개를 만들고 각각에 AI를 붙이면, **세 개의 작업이 동시에 진행됩니다.** 저는 그 사이를 오가며 방향을 잡고 우선순위를 조정하면 됩니다.

![[Pasted image 20260318081229.png]]

최근에는 Claude Code, Codex, Conductor, OpenCode 같은 도구들도 워크트리를 더 쉽게 다룰 수 있는 인터페이스를 제공하고 있습니다. 덕분에 예전처럼 터미널에서만 어렵게 다루지 않아도, 데스크톱 앱 환경에서 훨씬 편하게 병렬 작업을 구성할 수 있게 됐습니다.

![[스크린샷 2026-03-18 오전 8.22.44.png]]

그런데 여러 작업을 동시에 돌리기 시작하자, 곧 다른 문제가 보였습니다. 속도는 빨라졌지만 품질을 확인하는 일은 여전히 사람에게 남아 있었고, 그 검수 시간이 새로운 병목이 됐습니다.

## 멀티 에이전트로 코드 품질까지 같이 끌어올리기

처리량만 늘어난다고 생산성이 완성되지는 않습니다. AI가 만든 코드의 품질이 들쭉날쭉하다면, 결국 사람이 뒤에서 정리하고 리뷰하는 시간이 병목이 됩니다.

제가 겪어 본 바로는, AI 산출물의 품질이 흔들리는 이유는 대체로 두 가지였습니다.

첫째는 **정보가 너무 많이 섞이는 문제**였습니다. 흔히 컨텍스트 오염이라고 부르는 현상입니다. 작업에 꼭 필요하지 않은 정보가 너무 많이 섞이면 AI는 초점을 잃기 쉽습니다. 사람도 회의에서 관련 없는 이야기가 길어지면 핵심을 놓치는 것과 비슷합니다.

![[Pasted image 20260318083214.png]]

둘째는 **고치고 다시 확인하는 반복이 부족하다는 점**이었습니다. 피드백 루프가 약한 상태라고 표현할 수 있습니다. AI가 코드를 한 번 작성하고 바로 끝내 버리면 결과물의 편차가 커집니다. 사람도 초고를 바로 제출하면 품질이 안정적일 수 없는 것과 같은 이치입니다.

![[Pasted image 20260318083310.png]]

저는 이 두 문제를 멀티 에이전트 구조로 풀고 있습니다. 여기서 멀티 에이전트란, 하나의 AI가 모든 일을 떠안는 대신 역할이 다른 여러 AI가 팀처럼 움직이는 구조를 말합니다.

- **탐색 에이전트**는 관련 코드를 읽고 필요한 정보만 추려 옵니다.
- **구현 에이전트**는 그 정보를 바탕으로 코드를 작성합니다.
- **리뷰 에이전트**는 결과물을 검토하면서 코드 품질을 확인하고, 구조가 괜히 복잡해지지는 않았는지, 팀이 정한 방식에서 벗어나지는 않았는지, 테스트가 빠지지는 않았는지를 살펴봅니다.
- **관리 에이전트**는 어떤 에이전트를 언제 호출할지 결정하고, 전체 흐름을 조율합니다.

핵심은 리뷰 에이전트가 문제를 발견하면 구현 에이전트가 수정하고, 그 결과를 다시 리뷰하는 사이클이 자동으로 반복된다는 점입니다. 예전에는 제가 직접 코드를 읽고 “여기 다시 고쳐 주세요”라고 피드백했다면, 이제는 그 1차 수정과 검토 과정을 AI끼리 먼저 돌립니다.

![[멀티 에이전트 자동화.png]]

## 그래서 개발자의 일이 달라졌습니다

이 흐름 안에서, 요즘 제 업무에서 더 큰 비중을 차지하는 것은 직접 코드를 작성하는 일보다 계획을 점검하고 결과물을 검수하는 일입니다. AI가 제안한 작업 계획이 현재 맥락과 우선순위에 맞는지 확인하고, 구현 결과가 요구사항과 기존 코드의 규칙에 맞는지 최종적으로 살펴보는 일이 더 중요해졌습니다.

또 하나 달라진 점은, 개별 결과물만 보는 것이 아니라 AI가 일하는 환경 자체를 계속 조정하게 됐다는 점입니다. 비슷한 실수가 반복되면 그때그때 수정으로 끝내지 않고, 프롬프트와 리뷰 기준, 체크리스트, 자동화 규칙을 손봐서 같은 문제가 다시 나오지 않게 만듭니다. 한 번의 결과물을 고치는 것보다, 다음 결과물이 더 안정적으로 나오게 하는 데 시간을 더 쓰게 된 셈입니다.

![[Pasted image 20260319202349.png]]

일부에서는 이런 접근을 ‘하네스 엔지니어링’이라고 부르기도 합니다. 용어는 조금 낯설지만 뜻은 단순합니다. AI가 안정적으로 일할 수 있도록, 프롬프트뿐 아니라 문서와 규칙, 테스트, 로그, 피드백 루프까지 포함한 작업 환경 전체를 함께 설계하는 방식입니다.

## 마무리

요즘 저는 AI를 ‘대신 코드를 짜주는 도구’라기보다, 여러 작업을 나눠 맡기고 함께 진행하는 협업 도구에 가깝게 사용하고 있습니다. 특히 워크트리와 멀티 에이전트는, 한 번에 하나씩 처리하던 개발을 여러 갈래로 나눠 동시에 진행하게 해 준다는 점에서 인상적이었습니다.

물론 아직 정답이 정해진 방식은 아닙니다. 도구도 빠르게 바뀌고, 그에 따라 작업 방식도 계속 달라지고 있습니다. 다만 분명한 변화는 있습니다. 이제 중요한 것은 AI에게 얼마나 길게 설명하느냐가 아니라, 어떤 역할을 맡기고 어떤 구조 안에서 일하게 하느냐입니다.
