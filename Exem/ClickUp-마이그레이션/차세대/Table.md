## 주요 내용

### table 태그 vs div 태그
- 둘 사이 장단점이 있으며, 자세히 조사해보고 선택할 필요가 있음

### 테스트 전략

#### 유닛 테스트
- 한 번 작성된 이후로 수정되지 않는게 좋음
- 리팩토링 계획이 있고 많은 코드 수정이 예상되는 상황이라면, 버려지는 유틸리티 함수나 실패/제거되는 유닛 테스트가 발생할 수 있음
- 이런 상황이라면 차라리 **통합 테스트**가 더 가치 있음

#### 통합 테스트
- 세부 구현에 의존하는 테스트를 작성해서는 안됨
- 리팩토링을 수행했는데 테스트가 깨져버린다면 잘못 작성된 테스트
- 테스트를 그만 작성하게 되는 가장 큰 포인트

#### Vitest 브라우저 모드
- 최근 Vitest에서 나온 브라우저 모드 테스트가 아주 좋음

### 기능 관리
- 테이블 기능이 20개 가까이 되면서 각 기능과의 동작 호환 관리가 힘들었음
- **대칭표 기반**으로 관리하니 아주 편했음
- 스토리북이나 테스트 코드도 이런 형태로 관리하면 필요한 부분에만 필요한 테스트/문서 작성 가능

### Tanstack Table 인사이트
- Tanstack Table을 사용하고 있다면 Table 컴포넌트의 로컬 상태는 필요 없을 수 있음
- **제어/비제어** 관점의 설계 철학을 이해하는게 도움이 많이 됨
- **Custom Features** API: Tanstack Table의 내부 기능 동작 방식과 동일하며, 모든 기능을 오버라이드 할 수 있고, 새로운 기능을 구현해 Table 인스턴스 자체에 메서드/프로퍼티를 주입할 수 있음
